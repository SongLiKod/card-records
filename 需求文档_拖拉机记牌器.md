## 第一部分：拖拉机游戏规则分析

### 一、游戏概述

**拖拉机（又称"升级""80分""找朋友"）** 是中国最流行的扑克牌游戏之一。其核心玩法为**庄家 vs 闲家** 的对抗，通过出牌、抢庄、扣底等机制决定胜负。

### 二、牌具规格

| 玩家人数 | 使用牌数 | 总牌数 | 每人手牌数 |
|---------|---------|-------|-----------|
| 4 人    | 1 副（54张） | 54 张 | 12 张（底牌 6 张） |
| 4 人    | 2 副（108张）| 108 张| 25 张（底牌 8 张） |
| 6 人    | 3 副（162张）| 162 张| 25 张（底牌 12 张） |
| 8 人    | 4 副（216张）| 216 张| 25 张（底牌 16 张） |

> **注**：拖拉机也可用 3 副牌（6人）或 4 副牌（8人）进行，规则类似但牌数缩放。

### 三、牌型与大小

#### 3.1 牌力大小顺序（从大到小）

```
大王 > 小王 > 主牌级牌 > 副牌级牌 > 主花色A > K > Q > J > 10 > 9 > 8 > 7 > 6 > 5 > 4 > 3 > 2
```

- **级牌（当前级数的牌）**：如当前打 3，则所有 3 都是级牌
- **主牌**：由 大王、小王、级牌、主花色牌组成
- **副牌**：非主花色的其他牌

#### 3.2 拖拉机定义

**拖拉机**是指同花色中相邻级别（或同级别）至少两个对子的连续组合：

- 单张拖拉机（顺子）：如 ♠A♠K♠Q（连续 3 张以上）
- **对子拖拉机（核心）**：如 ♠AA♠KK♠QQ（连续 3 对以上，2 副牌以上才常见）
- **连对**：至少 2 对连续对子，如 ♠AA♠KK

**不同牌数的拖拉机判定差异：**

| 项目 | 1 副牌 | 2 副牌 | 3 副牌 |
|-----|-------|-------|-------|
| 拖拉机最小长度 | 3 张连续 | 4 张（2连对） | 6 张（3连对） |
| 常见拖拉机 | AA+K+Q 顺子 | 66+77+88 连对 | 666+777+888 三连拖 |
| 三张（三同） | 不存在 | 不存在 | ✅ 存在（2 张对子+1 张单） |

### 四、计分规则

**分数牌**是游戏的核心：

| 牌面 | 分值 |
|-----|------|
| 5 | 5 分 |
| 10 | 10 分 |
| K | 10 分 |

- **闲家**每轮吃到的分数累计，达到升级线即可**上台**
- **底牌被扣底**时，底牌分数 ×2（或更多，依规则）

### 五、基本流程

1. **发牌** → 2. **叫主/抢庄** → 3. **扣底牌** → 4. **出牌阶段** → 5. **计分/升级**

## 第二部分：需求文档 —— 拖拉机记牌器（Android）

### 1. 项目概述

**产品名称**：拖拉机记牌器（Tractor Card Tracker）  
**目标平台**：Android 5.0+  
**核心定位**：一款在拖拉机游戏进行时悬浮于屏幕之上的半透明记牌辅助工具，自动记录出牌信息、分析剩余牌况，帮助玩家做出更优决策。

### 2. 功能需求（Functional Requirements）

#### FR-1：悬浮窗功能（核心）

| 编号 | 需求描述 | 优先级 |
|-----|---------|-------|
| FR-1.1 | 应用启动后，在主游戏界面之上显示一个悬浮窗 | P0 |
| FR-1.2 | 悬浮窗默认半透明（透明度 30%~50%），不遮挡牌面 | P0 |
| FR-1.3 | 支持用户任意拖动悬浮窗位置，可放置于屏幕四角或任意位置 | P0 |
| FR-1.4 | 双击悬浮窗可切换展开/收起状态（展开时显示详细统计） | P1 |
| FR-1.5 | 悬浮窗尺寸自适应，支持不同屏幕分辨率 | P1 |
| FR-1.6 | 提供设置入口，用户可调节透明度（10%~80% 可调） | P2 |
| FR-1.7 | 支持"穿透模式"开关——开启后触摸不拦截，直接透传到底层游戏 | P2 |

#### FR-2：记牌功能

| 编号 | 需求描述 | 优先级 |
|-----|---------|-------|
| FR-2.1 | 记录所有已打出的牌，按花色和时间顺序排列 | P0 |
| FR-2.2 | 统计并显示每种花色剩余牌张数 | P0 |
| FR-2.3 | 统计并显示未出现的所有牌 | P0 |
| FR-2.4 | 高亮显示尚未出现的分数牌（5、10、K） | P0 |
| FR-2.5 | 剩余分数牌按"已出/未出"分类显示（如"未出 5×3、10×2、K×4"） | P0 |
| FR-2.6 | 按花色统计剩余分数分值（如♠还剩25分、♥还剩15分） | P1 |

#### FR-3：缺门分析

| 编号 | 需求描述 | 优先级 |
|-----|---------|-------|
| FR-3.1 | 分析并显示**自己**缺哪一色牌（全部打出或从未获得） | P0 |
| FR-3.2 | 分析并显示**各玩家**缺哪一色牌（通过出牌行为推断） | P0 |
| FR-3.3 | 当某玩家在某一花色上已出牌数达到该花色总牌数时，标记为"断门" | P0 |
| FR-3.4 | 用直观图标显示断门状态（如红色×、灰色◻） | P1 |
| FR-3.5 | 支持根据出牌行为推断对家及上下家断门 | P1 |

#### FR-4：自动数据获取

| 编号 | 需求描述 | 优先级 |
|-----|---------|-------|
| FR-4.1 | 应用在启动时自动捕获当前游戏画面数据，无需手动输入 | P0 |
| FR-4.2 | 支持通过**屏幕截图 + OCR 识别**自动读取已出牌信息 | P0 |
| FR-4.3 | 支持通过**无障碍服务（AccessibilityService）**监听游戏界面文字变化 | P1 |
| FR-4.4 | 支持用户手动修正识别的错误（长按某个牌进行编辑） | P2 |
| FR-4.5 | 自动识别游戏人数、当前级数、庄家位置等关键信息 | P1 |

#### FR-5：多牌局支持

| 编号 | 需求描述 | 优先级 |
|-----|---------|-------|
| FR-5.1 | 支持 4 人 1 副牌模式 | P0 |
| FR-5.2 | 支持 4 人 2 副牌模式 | P0 |
| FR-5.3 | 支持 6 人 3 副牌模式 | P1 |
| FR-5.4 | 支持 8 人 4 副牌模式 | P2 |
| FR-5.5 | 用户可手动切换牌局类型 | P1 |
| FR-5.6 | 新牌局开始时自动重置所有统计数据 | P1 |

#### FR-6：视觉与统计展示

| 编号 | 需求描述 | 优先级 |
|-----|---------|-------|
| FR-6.1 | 展开态悬浮窗分区域显示"已出牌""剩余牌""分数统计""断门分析" | P0 |
| FR-6.2 | 剩余牌用麻将牌式竖排布局，一目了然 | P1 |
| FR-6.3 | 分数统计用进度条/环形图展示（如"已吃 120 分 / 总分 200 分"） | P2 |
| FR-6.4 | 支持深色/浅色主题（跟随系统） | P2 |

### 4. 用户界面设计

#### 4.1 悬浮窗（收起态）

```
 ┌──────────────────────────────┐
 │  ♠ ♥ ♣ ♦                     │  ← 花色简略统计
 │  8 5 7 3  │ 剩 85分           │  ← 分数概览
 │  [≡]                      [×] │  ← 拖动手柄 / 关闭
 └──────────────────────────────┘
```

- **尺寸**：约 140 × 50 dp
- **背景**：半透明黑色（圆角 12dp）
- **内容**：四花色剩余张数 + 剩余总分

#### 4.2 展开态

```
 ┌──────────────────────────────────┐
 │  已出牌                          │
 │  ♠ A K Q J 10 9 8 ...           │  ← 灰色标识已出
 │  ♥ 10 9 7 6 ...                  │
 │  ♣ A K ...                       │
 │  ──────────────────────           │
 │  剩余牌                          │
 │  ♠ 6 5 4 3 2                     │  ← 高亮分数牌(5/10/K)
 │  ♥ K Q J 8 5 4 3 2              │
 │  ──────────────────────           │
 │  分数统计                        │
 │  未出：5×3(15分) 10×2(20分) K×4(40分) │
```

### 5. 技术方案建议

#### 5.1 悬浮窗实现

| 技术 | 说明 |
|-----|------|
| `WindowManager.LayoutParams.TYPE_APPLICATION_OVERLAY` | Android 6.0+ 推荐悬浮窗类型 |
| 使用 `Service` 作为常驻后台 | 保证悬浮窗在游戏切换时不消失 |
| `GestureDetector` 处理拖拽 | 实现平滑拖动效果 |
| `ValueAnimator` 展开/收起动画 | 提升交互体验 |

#### 5.2 自动识别方案

**方案 A：OCR 识别（推荐作为主要方案）**

游戏画面 -> 截图（MediaProjection API）-> 截取牌局区域 -> OCR识别牌面 -> 数据更新

| 组件 | 选择 | 说明 |
|-----|------|------|
| OCR 引擎 | Tesseract OCR Android（或 PaddleOCR Lite） | 支持中文字符识别 |
| 图像预处理 | OpenCV（自适应二值化、去噪） | 提高识别精度 |
| 截图方式 | MediaProjection API | 无需 root，需用户授权 |
| 文字检测 | 连通域分析 + 模板匹配 | 针对数字和花色符号优化 |

**方案 B：无障碍服务（AccessibilityService）**

监听界面文本变化 -> 提取牌面信息 -> 解析数据结构 -> 更新状态

- 适用于使用原生 UI 控件的游戏
- 不需要截图权限，更轻量
- 但只对基于 TextView/WebView 的游戏有效

**方案 C：屏幕截图 + 图像模板匹配（混合方案）**

- 对常见的拖拉机游戏 App 做针对性模板匹配
- 可预先训练牌面模板库
- 速度和准确率高但通用性略差

#### 5.3 数据模型

```json
{
  "gameType": "2_decks",
  "level": 3,
  "players": [
    { "id": 0, "position": "self", "name": "我" },
    { "id": 1, "position": "left", "name": "上家" },
    { "id": 2, "position": "opposite", "name": "对家" },
    { "id": 3, "position": "right", "name": "下家" }
  ],
  "playedCards": [
    { "suit": "spade", "rank": "A", "player": 0 },
    { "suit": "heart", "rank": "5", "player": 1 }
  ],
  "voidSuits": {
    "0": { "heart": true },
    "1": { "spade": true, "diamond": true }
  },
  "scoreRemaining": {
    "5": 3,
    "10": 2,
    "K": 4
  }
}
```

#### 5.4 断门判断算法

```
规则1：如果某玩家在花色X上已出牌张数 = 该花色总牌数，则该玩家"断门X"
规则2：如果某轮花色为X，某玩家没有出X花色牌（出了其他花色或主牌），则该玩家"断门X"
规则3：连续多轮未跟出某花色，累计置信度提升
规则4：对于1副牌，每花色13张+1张主牌级牌；2副牌每花色26张；以此类推
```

### 6. 交互流程

启动App -> 请求权限（悬浮窗权限、通知权限、截图权限）-> 选择牌局类型（默认2副牌）-> 浮动窗显示（半透明，可拖动）-> 等待截图/无障碍事件 -> 识别牌面 -> 更新数据 -> 更新悬浮窗显示 -> 断门分析更新

### 7. 发布与分发

| 阶段 | 内容 | 目标日期 |
|-----|------|---------|
| MVP | 纯手动输入版本（无自动识别），验证核心功能 | T+2 周 |
| V1.0 | 自动识别 + 核心记牌 + 断门分析 | T+6 周 |
| V1.1 | 多游戏兼容 + UI 优化 | T+10 周 |
| V1.2 | 用户反馈迭代 | T+14 周 |

### 8. 挑战与风险

| 风险 | 影响 | 缓解方案 |
|------|-----|---------|
| 不同游戏 App 牌面风格不同，OCR 准确率下降 | 高 | 采用可插拔的模板匹配 + OCR 混合方案 |
| Android 系统悬浮窗权限限制 | 中 | 提供详细的权限引导页面 |
| 游戏运行中截图性能开销 | 中 | 降低截图频率（200ms 一次）+ 异步识别 |
| 多副牌时信息量大，悬浮窗过拥挤 | 中 | 支持折叠/展开 + 字体自适应 |
| 游戏版本更新可能改变 UI 布局 | 高 | OCR识别 + 数据验证纠错机制 |
